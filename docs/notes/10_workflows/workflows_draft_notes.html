End-to-end workflows

The main steps we shouold go through when implementing a ML project are:

  1. Understand the big picture
  2. Get the data and do a preliminary exploration
  3. Create a representative test set and lock it away
  4. Explore the train data to gain insights
  5. Experiment with attirbute combinations
  6. Prepare train data for ML algorithms
  7. Try out different candidate models and estimate accuracy metrics with CV
  8. Go back to the drawing board if needed
  9. Select a model and fine-tune it
  10. Evaluate model on test set
  11. Present your results!

We’ll dive deeper into some of these steps.

Understand the big picture

This step is all about understanding the framing of your project, you should ask your team questions such as:

  • Clearly define the objective
  • How will the results be used?
  • What are the current solutions or workarounds (if any)?
  • How should you frame this problem? Classification/regression, supervised/unsupervised.
  • How will performance be measured in the context of the problem?
  • What would be the minimum performance needed to reach the objective?
  • What ML performance metric will be used to measure performance?
  • What are comparable problems? Can you reuse experience or tools?
  • Is human expertise available?
  • What assumptions have been made so far?
  • Verify the assumptions if possible.

For simplicity, for the rest of the section we will assume we are in the supervised scenario. But most steps also apply for unsupervised problems.

Get the data and do a preliminary exploration

After you get the data, we can open it and do some superficial exploration. If you are wroking with tabular data, this can be done using pandas.DataFrame methods such as head(), info(), value_counts(), describe(), and hist(). The goal here is to get a high-level understanding of the variables in your dataset and answer questions such as: - What is the data type of each variable? - Which are categorical and which numerical? - If this is a supervised problem, which is the target variable? - If there are target classes, are any of them rare? - Are there any variables with a lot of missing values? - Do the data distributions make sense? - Do you understand how each variable was computed?

Create a representative test set and lock it away

We have previously discussed the importance of keeping the test set unseen and locked away to prevent data leakage (inadvertently tuning our model to the test set). The test set should only be used to assess the final performance of our model and not to influence any modeling decisions. This way, the test set it is truly representative of how our model will perform on new, unseen data.

Typically, we set aside 20% or 30% of the data for testing, but if your dataset is massive a smaller percentage will probably be ok.

For the implementation, we usually use the function train_test_split from sklearn. Setting a random_state in this function ensures that we can always reproduce the same train-test split. If there are rare target classes or you want certain features to be proportionally represented in the test set, you can indicate so through the stratify parameter.

Explore the train data to gain insights

The main goals for this step are to (1) identify any transformations or cleaning that may be needed for the data and (2) identify any promising predictors, and (3) brainstorm if there are additional features we may want to compute.

For each variable, we should start by studying its characteristics:

  • Name
  • Type (categorical, int/float, bounded/unbounded, text, structured, etc.)
  • % of missing values
  • Noisiness and type of noise (stochastic, outliers, rounding errors, etc.)
  • Possibly useful for the task?
  • Type of distribution (Gaussian, uniform, logarithmic, etc.)

If there are not too many predictors, it can be useful to plot each one (or at least the most promising ones) against the target variable. This may uncover noise or errors in the data and iluminate whether data cleaning or transformation is needed. Plots may also show strong correlations that can give us an idea of which predictors might be best to keep.

In addition, we may also want to study the correlations between the predictors and target variables using the corr() method. The correlation coefficient ranges from -1 to 1 and

  • when it is close to 1, there is a strong positive linear correlation,
  • when it is close to -1, there is a strong negative linear correlation.
  • when it is close to 0, there is no linear correlation.

Remember the correlation here is linear! There may be other kinds of correlations that may not be captured.

Generally, two things can go wrong when we implement an ML model: “bad algorithm” or “bad data”. Let’s see some of the challenges that you can encounter in these two categories:

Garbage in, garbage out

Training dataset is…

  • too small: It is not always easy or cheap to get extra training data but always, but generally more high-quality training data will make your models perform better. Potential solution: collect more high-quality training data.
  • nonrepresentative: the training data must be representative of the cases you want to generalize to. If the sample is too small, you will have sampling noise, but even if it’s large, if it’s not representative you will have sampling bias. Potential solutions: if possible, collect more data with better sampling.
  • poor quality: has too many outliers, errors, and missing data, making it harder for the algorithm to detect the underlying pattern. Potential solutions: remove outliers or correct errors, for observations missing features, decide whether to remove these features, remove the observations, fill in the missing value, or compare multiple approaches.
  • full of irrelevant features: irrelevant features can add too much noise for the model to work well. Potential solutions: select the most useful features to train on among existing ones (feature selection); combine existing features to produce more useful ones (feature extraction); gather new data to add features

Problems with algorithms

  • Overfitting: when the model starts detecting patterns in the noise itself, fitting the trainiing data too closely and resulting in poor generalization.

Potential solutions: - choose a less flexible model (e.g. a linear model instead of a high-degree polynomial) - Collect more training data - Reduce the noise in the training data - Adjust the hyperparameters in the model to reduce variance

  • Underfitting: when the model is too rigid or simple to detect the structure of the data:

Potential solutions: - choose a more flexible model - Add better features to you data - Adjust the hyperparameters in the model to allow more variance
